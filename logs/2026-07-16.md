## Stripe webhook QA pass + reliability hardening
- New `scripts/test-payment-webhook-flow.mjs` — fabricates signed Stripe webhook events and POSTs them at the local dev server to exercise `/api/stripe/webhook` end-to-end (no real Checkout/browser session needed). Covers `checkout.session.completed` (+ duplicate-delivery idempotency), `checkout.session.expired`, `charge.failed`, `charge.refunded`, `charge.dispute.created`, invoice auto-paid linking. Cleans up all rows it creates.
- Fixed bug in `src/app/api/stripe/webhook/route.ts`'s `markPaymentPaid`: a duplicate webhook delivery (Stripe is at-least-once, not exactly-once) used to overwrite `paid_at` with a later timestamp every redelivery. Now no-ops once already paid.
- Fixed a data-integrity bug in the same function: marking a payment paid unconditionally overwrote the linked invoice's `signed_at`, which could stomp a REAL client e-signature timestamp/`signer_name` (from today's earlier e-sign feature) with the payment-completion time. Now only backfills `signed_at` if the invoice was never actually signed.
- Hardened the `account.updated` handler: `connect_onboarded` is now only updated when `getRecipientTransferStatus` returns a concrete status — an ambiguous/unreadable response no longer forces a working contractor's account offline.
- Added a `charge.dispute.created` handler that logs chargebacks server-side (`[DISPUTE]` lines) — previously completely unhandled/invisible. No new payment status added (would need a schema migration); logging only for now.
- Updated `TEST_PLAN.md`: added an "Automated verification" section documenting the script, refreshed the stale "Known Limitations" (invoice signature + PDF attachment + fee transparency were already resolved in earlier work but the doc wasn't updated), and checked off the summary checklist items actually verified this pass.
